🗑️ Garbage Collector

⚠️ GC Pause Time

  • <10ms: Ótimo - Nenhuma ação
  • 10-50ms: Aceitável - Monitorar
  • 50-100ms: Atenção - Documentar e informar equipe

⚠️ Heap Size

  • Crescimento gradual e estabilização: Normal
  • Crescimento contínuo sem estabilizar: Possível memory leak - Documentar e informar N2

🟠 Outros Incidentes Importantes

Procedimento P2: Degradação de Performance (SLO Violado)

  1. Notifique N2: "⚠️ SLO de latência violado - P99 fora do alvo"
  2. Colete: Valor P99 atual, P95, P90, P50
  3. Use filtro de Endpoint para identificar qual está lento
  4. Verifique "Endpoints Mais Lentos" - identifique o pior
  5. Verifique GC Pause Time - se >100ms, pode ser a causa
  6. Verifique ThreadPool - threads disponíveis?
  7. Reporte informações completas para N2
  8. Aguarde orientação técnica

Procedimento P3: Anomalia de Tráfego

  1. Identifique o tipo de anomalia: Pico súbito ou queda abrupta?
  2. Verifique se há manutenção programada ou evento esperado
  3. Se pico inesperado: Verifique Taxa de Erro (pode estar relacionado)
  4. Se queda inesperada: Verifique Kestrel conexões rejeitadas
  5. Documente e informe N2
  6. Se variação >100% do normal, trate como incidente

Procedimento P5: Problema de Memória

  1. Notifique N2: "⚠️ Possível problema de memória - GC anormal"
  2. Colete: GC Pause Time, Heap Size, Gen2 Collections
  3. Tire screenshot dos gráficos de GC
  4. Verifique se heap está crescendo continuamente
  5. Possível memory leak - escalação para N2/N3 necessária
  6. Pode ser necessário reinício programado

❓ Perguntas Frequentes (FAQ)

Q: Posso sair para almoço/intervalo?

A: Sim, mas NUNCA deixe o dashboard sem monitoramento. Coordene com outro operador ou peça cobertura. Em caso de incidente ativo, aguarde resolução ou passagem adequada antes de sair.

Q: E se eu não souber o que fazer?

A: SEMPRE escalone para N2 quando tiver dúvidas. É melhor escalar sem necessidade do que tomar ação errada ou deixar problema sem atendimento.

Q: Posso reiniciar a aplicação sem autorização?

A: NÃO! NUNCA! Reinícios causam indisponibilidade e podem piorar o problema. Sempre requer autorização de N3.

Q: O que fazer se perder acesso ao dashboard?

A: 1) Verifique sua conexão de internet, 2) Tente acessar de outro navegador/dispositivo, 3) Se persistir, notifique N2 IMEDIATAMENTE e registre no log que está sem monitoramento.

Q: Como sei se é um ataque DDoS?

A: Throughput >500% do normal + Padrão de requisições anormal + Conexões rejeitadas. Se suspeitar, escalone para N2 e Segurança.

Q: O dashboard mostra tudo verde, mas usuários reclamam de lentidão

A: 1) Verifique se filtros (Instance, Endpoint, Environment) estão corretos e mostrando todos os dados, 2) Verifique métricas de percentis (P95, P99), 3) Use a filtragem por Endpoint para ver SLOs específicos, 4) Dashboard pode estar agregando dados de múltiplas instâncias. Reporte ao N2.

Q: O que significam os novos gráficos de operações por hora?

A: São métricas de negócio que mostram quantas operações CRUD (Create, Read, Update, Delete) acontecem em cada hora para cada entidade. Ajudam a identificar padrões de uso e anomalias como exclusões em massa ou picos anormais de leitura.

Q: Quando devo me preocupar com as métricas CRUD?

A: Preocupe-se quando: 1) Pico anormal de Deletes (pode ser bug ou ação maliciosa), 2) Reads muito acima do normal (possível scraping), 3) Creates zerados em horário comercial (sistema pode estar com problema), 4) Qualquer padrão muito diferente dos dias anteriores no mesmo horário.

Q: Esqueci de documentar um incidente, e agora?

A: Documente assim que lembrar, mesmo que tardiamente. Melhor documentar tarde do que não documentar. Inclua nota de que documentação é retroativa.

Q: Posso modificar thresholds ou configurações do dashboard?

A: NÃO! Qualquer mudança em thresholds ou configurações deve ser aprovada pela equipe técnica e documentada formalmente.

📚 Glossário de Termos

Termo Significado
API Application Programming Interface - Interface de Programação de Aplicações
CRUD Create, Read, Update, Delete - As 4 operações básicas de dados
Throughput Volume de requisições processadas por segundo (req/s)
Latência Tempo de resposta de uma requisição
P50/P90/P95/P99 Percentis - Indicam que X% das requisições são mais rápidas que este valor
SLO Service Level Objective - Objetivo de nível de serviço
SLA Service Level Agreement - Acordo de nível de serviço (contrato)
4xx Códigos de erro do cliente (400, 401, 403, 404, etc)
5xx Códigos de erro do servidor (500, 502, 503, 504, etc)
ThreadPool Pool de threads gerenciadas pelo .NET para processar requisições
GC (Garbage Collector) Coletor de lixo - Gerenciador automático de memória do .NET
Heap Área de memória alocada pela aplicação
Kestrel Servidor web do .NET Core/ASP.NET Core
Deploy Implantação de nova versão da aplicação
Rollback Voltar para versão anterior da aplicação
DDoS Distributed Denial of Service - Ataque de negação de serviço distribuído
Memory Leak Vazamento de memória - Memória alocada não sendo liberada
Annotation Marcação no gráfico indicando um evento (deploy, alerta, etc)
Endpoint Rota específica da API (ex: /api/users, /api/orders)
Instance Cópia individual da aplicação em execução (pode haver múltiplas)
Prometheus Sistema de monitoramento e time-series database
Timeseries Gráfico que mostra valores ao longo do tempo
