📈 Painéis de Operações por Hora

📊 Operações de Registros por Hora

O que mostra: Gráfico timeseries com a quantidade de operações CRUD realizadas em cada hora, agrupadas por tipo de operação.

Métricas utilizadas:

# Created sum by (hour) (increase({entity}_records_created_by_hour_total[5m])) # Updated sum by (hour) (increase({entity}_records_updated_by_hour_total[5m])) # Deleted sum by (hour) (increase({entity}_records_deleted_by_hour_total[5m])) # Read sum by (hour) (increase({entity}_records_read_by_hour_total[5m]))

Informações na Legenda:

  • Last: Último valor registrado
  • Max: Valor máximo no período
  • Mean: Média no período

🔍 Como Interpretar os Gráficos CRUD

✅ Padrões Normais

  • 📈 Reads concentrados em horário comercial: Usuários consultando dados durante o expediente
  • 📊 Creates distribuídos ao longo do dia: Cadastros seguindo o fluxo de trabalho
  • 🔄 Updates após Creates: Dados sendo corrigidos/complementados após cadastro
  • 📉 Deletes baixos e estáveis: Exclusões são normalmente raras

⚠️ Padrões de Alerta

Padrão Possível Causa Severidade
Pico súbito de Deletes Exclusão em massa / Bug / Ação maliciosa 🔴 CRÍTICO
Reads >500% do normal Scraping / Loop infinito / Ataque 🟠 ALTO
Creates zerados em horário comercial Sistema travado / Erro de integração 🟠 ALTO
Updates repetitivos no mesmo registro Bug de loop / Retry excessivo 🟡 MÉDIO

📊 Métricas Complementares por Controller

Para cada entidade, também estão disponíveis:

  • Requisições (req/s): Taxa de requisições por segundo para o controller
  • Duração P95 (ms): Percentil 95 do tempo de resposta
  • Registros por Resposta (média): Quantidade média de registros retornados por requisição
💡 Dica de Uso: Use os gráficos CRUD para identificar horários de pico de cada entidade. Isso ajuda a planejar manutenções e entender o padrão de uso do sistema.

🌐 Métricas HTTP Detalhadas

Análise aprofundada do tráfego HTTP da API.

Latência por Percentis

⏱️ P50 / P90 / P95 / P99

O que são percentis:

  • P50 (Mediana): 50% das requisições são mais rápidas que este valor
  • P90: 90% das requisições são mais rápidas que este valor
  • P95: 95% das requisições são mais rápidas que este valor
  • P99: 99% das requisições são mais rápidas que este valor
💡 Como Interpretar:
  • Se P50 e P90 são baixos, mas P99 é alto: poucos usuários têm experiência ruim
  • Se todos os percentis são altos: problema generalizado de performance
  • Se há picos súbitos: pode indicar problemas intermitentes ou carga pesada

Taxa de Requisições por Status

📈 Status HTTP

Categoria Códigos Significado Ação
2xx - Sucesso 200, 201, 204 Requisições processadas com sucesso ✅ Normal - Continue monitorando
4xx - Erro Cliente 400, 401, 403, 404 Erro na requisição do cliente ⚠️ Verifique validações e autenticação
5xx - Erro Servidor 500, 502, 503, 504 Erro interno da aplicação 🚨 Investigação urgente necessária

Top 10 Endpoints

🐌 Endpoints Mais Lentos

O que mostra: Os 10 endpoints com maior tempo de resposta (P95).

Como usar:

  • Identifique gargalos de performance
  • Priorize otimizações baseado no impacto
  • Compare com volume de requisições para avaliar criticidade

💥 Endpoints com Mais Erros

O que mostra: Os 10 endpoints com maior taxa de erro.

Como usar:

  • Identifique endpoints problemáticos
  • Correlacione com a seção de exceções
  • Verifique se há problemas de validação ou dependências externas

⚙️ Runtime .NET & Garbage Collector

Monitoramento da saúde interna da aplicação .NET e gerenciamento de memória.

Garbage Collector (GC)

🗑️ GC Collections por Geração

Como funciona o GC do .NET:

  • Gen0: Objetos de curta duração (criados e destruídos rapidamente)
  • Gen1: Objetos de média duração (sobrevivem algumas coletas)
  • Gen2: Objetos de longa duração (permanecem na memória)
⚠️ Sinais de Alerta:
  • Gen0 muito alto: Normal em APIs com alto tráfego
  • Gen1 crescente: Objetos ficando mais tempo na memória
  • Gen2 frequente: Possível memory leak ou objetos grandes não liberados

💾 Heap Size & Working Set

Heap Size: Memória total alocada pelo runtime .NET.

Working Set: Memória física utilizada pelo processo.

📈 Padrão Saudável:
  • Heap size cresce gradualmente e estabiliza
  • Após GC Gen2, heap size deve reduzir
  • Se heap cresce continuamente: memory leak provável

🧵 ThreadPool .NET

O ThreadPool gerencia as threads usadas para processar requisições. É crítico para a capacidade da aplicação.

🔢 Thread Pool

O que mostra: Threads disponíveis para processar novas requisições.

🚨 CRÍTICO - Available Threads próximo de zero:
  • Novas requisições ficarão em fila
  • Timeouts começarão a ocorrer
  • API pode parecer "travada"
Causas comuns:
  • Código síncrono bloqueando threads (sem async/await)
  • Chamadas longas para serviços externos
  • Deadlocks ou operações travadas
  • Carga acima da capacidade
💡 Boas Práticas para ThreadPool:
  • Use async/await em toda operação I/O
  • Evite código síncrono que bloqueia threads
  • Configure timeouts apropriados para chamadas externas
  • Use HttpClient compartilhado (singleton)
  • Considere circuit breakers para serviços externos

❌ Exceções .NET

Rastreamento detalhado de todas as exceções não tratadas na aplicação.

📊 Exceções por Tipo

O que mostra: Contador e taxa de exceções ao longo do tempo, separadas por tipo.

⚠️ Tipos Comuns de Exceção:
Tipo Causa Provável Ação
NullReferenceException Tentativa de acessar objeto null Adicionar validações e null checks
TimeoutException Operação demorou muito Revisar timeouts e performance
HttpRequestException Falha ao chamar API externa Implementar retry e circuit breaker
SqlException Erro no banco de dados Verificar queries e conexões
UnauthorizedAccessException Problema de permissão Revisar autenticação/autorização

🌐 Kestrel Web Server

Métricas do servidor web integrado do .NET (Kestrel), responsável por gerenciar conexões HTTP.

🔌 Conexões Ativas

O que mostra: Conexões atualmente estabelecidas com o servidor.

🟢 0-200
Normal
🟡 200-500
Moderado
🟠 500-1000
Alto
🔴 >1000
Crítico

⏱️ Duração de Conexão P95

O que mede: Quanto tempo 95% das conexões permanecem abertas.

🟢 <30s
Normal
🟡 30-60s
Atenção
🔴 >60s
Problema
⚠️ Duração muito alta pode indicar:
  • WebSockets ou long-polling ativo
  • Clientes não fechando conexões (sem Connection: close)
  • Keep-alive muito longo

📋 Conexões Enfileiradas

O que são: Conexões aguardando para serem aceitas pelo servidor.

Ideal: Zero. Valores > 0 indicam que o servidor não consegue aceitar conexões rapidamente.

🔥 Se Conexões Enfileiradas está alto:
  • Servidor está sobrecarregado
  • ThreadPool pode estar saturado
  • Considere scaling horizontal
  • Revise configurações do Kestrel (MaxConcurrentConnections)

🚨 Sistema de Alertas Automáticos

O dashboard possui 4 alertas automáticos que aparecem como anotações no gráfico quando são acionados.

🚀 Deploy / Restart (Vermelho)

Condição: changes(dotnet_assembly_count[1m]) > 0

O que detecta: Quando a aplicação é reiniciada ou recebe um deploy.

Por que importa:

  • Marca no timeline quando mudanças foram aplicadas
  • Ajuda a correlacionar problemas com deploys
  • Identifica reinícios inesperados (crashes)

⚠️ Pico de Exceções (Laranja)

Condição: rate(dotnet_exceptions_total[1m]) > 1

O que detecta: Mais de 1 exceção por segundo.

Ação:

  • Verifique a seção "Exceções .NET" para ver o tipo
  • Correlacione com "Top 10 Endpoints com Erros"
  • Investigue logs detalhados da aplicação

🔥 Incidente - Alta Taxa de Erro (Vermelho)

Condição: Taxa de erro (5xx) > 5%

O que detecta: Falhas críticas afetando mais de 5% dos usuários.

Ação Imediata:

  • 🚨 Incidente crítico - Mobilize a equipe
  • Verifique se há deploy recente (annotation)
  • Analise endpoints com erro
  • Verifique dependências externas (banco, APIs)
  • Considere rollback se relacionado a deploy
🔥 Este é um alerta de SEVERIDADE MÁXIMA!
Indica que sua API está retornando muitos erros 5xx (erro do servidor), impactando significativamente os usuários.

📋 Guia de Uso Diário

Como usar este dashboard efetivamente no seu dia a dia.

✅ Checagem Rápida (30 segundos)

Use esta checagem para verificação rápida da saúde da API:

  1. 📊 Taxa de Erro está verde (<1%)?
  2. 🎯 SLOs estão sendo cumpridos (verde/amarelo)?
  3. 🚀 Throughput está normal para o horário atual?
  4. 🚨 Há alertas ativos no timeline?
  5. 🧩 Operações CRUD estão normais?

Se tudo está verde: ✅ Sistema saudável, continue monitorando.

Se algo está vermelho: 🔴 Prossiga para análise detalhada.

🔍 Análise de Performance (5 minutos)

Para investigar problemas de performance:

  1. Verifique Latência por Percentis:
    • P50 alto? Problema generalizado
    • P99 alto mas P50 normal? Problema em casos específicos
  2. Use filtro de Endpoint para isolar problemas
  3. Identifique Endpoints Lentos:
    • Veja "Top 10 Endpoints Mais Lentos"
    • Priorize aqueles com alto volume
  4. Verifique ThreadPool:
    • Available Threads baixo? Pode causar lentidão
    • Queue Length alto? Requisições estão enfileirando
  5. Analise GC:
    • GC Pause Time alto? Causa latência
    • Gen2 frequente? Possível memory leak

🧩 Análise de Métricas de Negócio (5 minutos)

Para entender padrões de uso e identificar anomalias:

  1. Revise os gráficos CRUD por Hora:
    • Os padrões estão consistentes com dias anteriores?
    • Há picos ou quedas anormais?
  2. Verifique Deletes:
    • Qualquer pico de deletes deve ser investigado
    • Deletes acima do normal podem indicar problemas
  3. Analise Reads:
    • Reads muito acima do normal podem indicar scraping ou loop
    • Reads zerados podem indicar sistema travado
  4. Compare entidades:
    • Use os valores de last/max/mean na legenda
    • Identifique quais entidades têm mais atividade

🔧 Investigação de Problemas

Se Taxa de Erro Aumentou:

  1. Veja "Taxa de Requisições por Status" - Qual status cresceu? (4xx ou 5xx)
  2. Consulte "Top 10 Endpoints com Erros"
  3. Use filtro de Endpoint nos SLOs
  4. Analise "Exceções .NET" - Qual tipo está ocorrendo?
  5. Verifique se há annotation de Deploy recente
  6. Consulte logs detalhados da aplicação

Se Latência Aumentou:

  1. Cheque "GC Pause Time" - GC está causando pausas?
  2. Verifique "ThreadPool Available" - Threads disponíveis?
  3. Analise "Endpoints Mais Lentos" - Qual endpoint degradou?
  4. Veja "Kestrel Conexões Enfileiradas" - Servidor sobrecarregado?
  5. Correlacione com throughput - Aumento de carga?

Se Throughput Caiu:

  1. Verifique "Kestrel Conexões Rejeitadas" - Servidor recusando?
  2. Analise "Taxa de Erro" - Usuários recebendo erros?
  3. Veja annotations - Houve deploy/restart?
  4. Pode ser queda natural no tráfego (horário)

⚠️ Sinais de Alerta Importantes

🚨 ATENÇÃO IMEDIATA NECESSÁRIA:
  • 🔴 Gen2 GC crescendo continuamente → Provável memory leak
  • 🔴 ThreadPool Available próximo de 0 → Aplicação vai travar
  • 🔴 Conexões Rejeitadas no Kestrel → Servidor sobrecarregado
  • 🔴 P99 latência crescente → Performance degradando
  • 🔴 Taxa de exceções aumentando → Bugs ou problemas externos
  • 🔴 SLO Success Rate abaixo de 95% → Violação crítica
  • 🔴 Pico anormal de Deletes → Possível exclusão em massa

📊 Análise de Tendências

Use o seletor de tempo (canto superior direito) para análise histórica:

  • Last 6 hours: Monitoramento em tempo real
  • Last 24 hours: Padrões diários e horários de pico

Perguntas a fazer:

  • Performance degrada durante picos?

⚙️ Variáveis Configuráveis

O dashboard possui várias variáveis que permitem filtrar e personalizar a visualização.

📊 Datasource
Seleciona qual servidor Prometheus usar como fonte de dados. Útil se você tem múltiplos ambientes (prod, staging).
🚀 Job/Application
Seleciona qual aplicação monitorar. Neste caso, filtra para jobs contendo "geradv-api". Suporta multi-seleção.
💻 Instance

🎯 Endpoint APRIMORADO

🌍 Environment
Filtra por ambiente (production, staging, development). Separa métricas por ambiente.
⏱️ SLO Latência P99
Define o objetivo de latência P99:
  • 0.5s - APIs ultra-rápidas
  • 1s - Padrão web responsivo
  • 2s - Processamento moderado (padrão)
  • 5s - Processamento pesado
✅ SLO Success Rate %
Define o objetivo de taxa de sucesso:
  • 95% - APIs não-críticas
  • 99% - Padrão de produção
  • 99.9% - Críticas de negócio (padrão)
  • 99.99% - Mission-critical
💡 Dica de Uso: Configure os SLOs de acordo com os requisitos do seu negócio e da criticidade da API. Revise periodicamente se os SLOs estão adequados ou muito permissivos/restritivos. Use o filtro de Endpoint para verificar SLOs de endpoints críticos individualmente.

✨ Boas Práticas de Observabilidade

📈 Cultura de Observabilidade

  1. Monitore Proativamente: Não espere problemas acontecerem. Revise o dashboard diariamente.
  2. Estabeleça Baselines: Conheça o comportamento "normal" da sua API em diferentes horários.
  3. Correlacione Eventos: Use annotations para marcar deploys e correlacionar com mudanças nas métricas.
  4. Defina SLOs Realistas: Baseie-se em dados reais, não em expectativas irrealistas.
  5. Automatize Alertas: Use as annotations como gatilho para notificações (Slack, PagerDuty, etc).
  6. Monitore o Negócio: Use os painéis CRUD para entender padrões de uso.

🎯 Foco nos Indicadores Certos

Os 4 Golden Signals (Google SRE):

  1. Latência: Quanto tempo leva para responder? → P50, P95, P99
  2. Tráfego: Quanto uso está recebendo? → Throughput (req/s)
  3. Erros: Qual a taxa de falha? → Taxa de Erro %
  4. Saturação: Quão "cheia" está a aplicação? → ThreadPool, Conexões

Este dashboard implementa todos os 4 sinais dourados! ✅

Métricas de negócio para entender o uso real do sistema! 🎯

🔄 Processo de Incident Response

  1. Detecção: Dashboard mostra alerta vermelho
  2. Triagem: Use checklist de investigação acima
  3. Diagnóstico: Correlacione múltiplas métricas
  4. Mitigação: Tome ação baseada no diagnóstico
  5. Documentação: Anote o que aconteceu e como resolveu
  6. Post-mortem: Analise causa raiz e previna recorrência

  • SLO (Service Level Objective): Meta para o SLI (ex: P99 < 2s em 99.9% do tempo)
  • SLA (Service Level Agreement): Contrato com consequências se SLO não for cumprido
  • Error Budget: Quanto "erro" é tolerável antes de violar o SLO
  • CRUD: Create, Read, Update, Delete - operações básicas de dados
